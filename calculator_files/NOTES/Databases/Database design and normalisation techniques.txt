# Database design and normalisation techniques

## Why Normalisation Is Used

- process of organising a database to reduce unnecessary duplication of data
- improves consistency of data
- improves integrity of data
- reduces anomalies
- produces a more efficient database structure


## First Normal Form (1NF)

### Requirements

- data must be atomic
- no repeating groups
- each field contains a single value

### Atomic Data

- Atomic data refers to a value that the database treats as a single unit rather than something that should be split into multiple values for the intended use
- examples that violate 1NF:
    - multiple phone numbers in one field
    - multiple products in one field




## Second Normal Form (2NF)

### Requirements

- everything in 1NF
- every non-key attribute must be dependent on the WHOLE of the primary key
- removes partial dependencies

### Important Note

- mainly applies to tables with composite primary keys
- if a table has a single-attribute primary key and is already in 1NF, it is automatically in 2NF

### Partial Dependency

- occurs when a non-key attribute depends on only part of a composite primary key
- these dependencies must be removed to achieve 2NF




## Third Normal Form (3NF)

### Requirements

- everything in 2NF
- non-key attributes are functionally independent
- removes transitive dependencies

### Functional Independence

- non-key attributes must depend only on the primary key
- non-key attributes must not depend on other non-key attributes

### Transitive Dependency

- occurs when a non-key attribute depends on another non-key attribute
- these dependencies must be removed to achieve 3NF







## When Is A Database Fully Normalised?

- when all relations are in 3NF
- data is atomic
- no partial dependencies exist
- no transitive dependencies exist









## 'Why Are Databases Normalised?' (PPQ)

### Data Duplication

- minimises data duplication
- reduces unnecessary repeated data
- do NOT say eliminates duplication

### Consistency

- eliminates data inconsistency
- improves consistency of stored data
- prevents multiple copies of the same data having different values

### Update Anomalies

- updates only need to be made in one location
- prevents different copies of the same data becoming inconsistent after updates

### Insertion Anomalies

- allows data to be added without requiring unrelated data to exist first

### Deletion Anomalies

- prevents deletion of one record accidentally removing important information about another entity









## 'Describe Two Problems That Can Occur With Databases That Are Not Fully Normalised' (PPQ)

### Data Duplication

- redundant/duplicated data wastes storage space

### Data Inconsistency

- if data is stored more than once then different copies may contain different values

### Update Problems

- if data is stored more than once then every copy must be updated when the value changes
- failure to update every copy causes inconsistency

### Insertion Problems

- it may not be possible to store data about one type of entity without creating a record for another type of entity

### Deletion Problems

- deleting one record may unintentionally remove data about another entity
- it may not be possible to delete one piece of information without deleting another

### Atomicity Problems

- difficult to select/edit/query data if fields are not atomic
- repeating groups make searching and updating more difficult










## Common Exam Definitions

### Primary Key

- attribute that uniquely identifies each record

### Composite Primary Key

- primary key made from two or more attributes

### Non-key Attribute

- attribute that is not part of the primary key

### Functional Dependency

- one attribute determines the value of another attribute

### Partial Dependency

- non-key attribute depends on only part of a composite primary key

### Transitive Dependency

- non-key attribute depends on another non-key attribute rather than directly on the primary key
