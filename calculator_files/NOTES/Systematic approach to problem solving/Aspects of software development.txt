# Systematic Approach to Problem Solving

## Analysis

### Purpose

- first stage of software development
- problem must be clearly defined before it can be solved
- requirements of the system must be identified
- requirements gathered through interaction with intended users
- aims to understand exactly what the system must do

### Activities

- define the problem
- identify user requirements
- identify inputs, outputs and processing requirements
- identify constraints
- create a data model
- determine success criteria

### Gathering Requirements

- interviews
- questionnaires
- observation
- meetings with users
- examination of existing systems

### Prototyping

- an early version of the system may be created
- helps clarify user requirements
- allows users to provide feedback
- requirements can be refined before development continues

### Agile Approach

- requirements may evolve during development
- regular user feedback is collected
- development is carried out in small iterations
- changes can be incorporated throughout the project








## Design

### Purpose

- specifies how the solution will be built
- acts as a blueprint for implementation
- completed before major development begins

### Activities

- design data structures for the data model
- design algorithms
- design program modules
- design subroutines and interfaces
- design the user interface
- design inputs, outputs and validation rules

### Modular Design

- solution divided into smaller modules
- each module performs a specific task
- improves maintainability
- improves readability
- makes testing easier
- allows collaborative development

### User Interface Design

- layout of screens/forms
- navigation structure
- accessibility considerations
- consistency of controls
- user experience

### Iterative Design

- design is often revised multiple times
- user feedback may cause design changes
- common in prototyping and agile approaches







## Implementation

### Purpose

- converts the design into a working solution
- models and algorithms are translated into data structures and code

### Activities

- writing source code
- creating data structures
- implementing algorithms
- integrating modules
- debugging errors

### Data Structures

- arrays
- records
- files
- stacks
- queues
- trees
- graphs
- hash tables

### Agile Implementation

- developed in iterations
- small sections completed and tested frequently
- user feedback influences future iterations

### Critical Path

- focus is placed on the most important features first
- core functionality implemented before less important features
- ensures the system remains usable even if time is limited







## Testing

### Purpose

- identifies errors and faults in the implementation
- verifies that the system behaves as expected
- checks whether requirements have been met

### Types of Test Data

#### Normal Data

- valid data
- expected/typical inputs
- should be accepted by the system

examples:
- age = 25
- mark = 60

#### Boundary Data

- values at the limits of accepted ranges
- used to test boundary conditions

examples:
- minimum valid value
- maximum valid value

#### Erroneous Data

- invalid data
- should be rejected or handled appropriately

examples:
- letters entered into a numeric field
- values outside permitted ranges

### Testing Activities

- unit testing
- integration testing
- system testing
- debugging

### Acceptance Testing

- carried out by intended users
- checks whether the system meets requirements
- determines whether the solution is acceptable for use
- confirms the system meets its specification









## Evaluation

### Purpose

- assesses the effectiveness of the completed system
- determines whether requirements have been met
- identifies possible improvements

### Evaluation Criteria

#### Functional Requirements

- does the system perform all required tasks?
- does it meet the specification?

#### Efficiency

- execution speed
- memory usage
- storage usage
- processor usage

#### Reliability

- operates correctly over time
- handles errors appropriately
- produces consistent results

#### Usability

- easy to learn
- easy to use
- intuitive interface
- accessible to intended users

#### Maintainability

- easy to modify
- easy to debug
- easy to update

#### Robustness

- handles invalid input correctly
- does not crash unexpectedly

#### Security

- protects data
- prevents unauthorised access
- enforces permissions

#### User Feedback

- feedback gathered from intended users
- determines satisfaction with the system
- identifies improvements for future versions






### Common Evaluation Questions

- were all requirements met?
- did the system solve the original problem?
- is the system efficient?
- is the system reliable?
- is the system easy to use?
- what improvements could be made?
